🔴 Alta Severidad

POST /v2/statements/sync — Sync masivo

No usa can ni policy. Cualquier usuario autenticado puede hacer un sync masivo de statements, sin verificar statements.sync.

Archivo
routes/api/v2.php → SyncStatementsController
Fix
Agregar can:sync,Statement::class o verificar hasPermissionTo(StatementsSync) en el controller/request.

PUT /v2/calculations/{calculation} — Editar cálculo

No tiene can ni policy. Solo requiere auth. El permiso calculations.edit existe pero nunca se verifica.

Archivo
routes/api/v2.php
Fix
Crear CalculationPolicy::update() que verifique hasPermissionTo(CalculationsEdit).

DELETE y POST /v2/calculations/* — CRUD cálculos

Misma situación que edit — ninguna ruta de cálculos valida permisos más allá de auth.

Afecta
DELETE /calculations/{calculation}, POST /calculations/
Fix
Implementar CalculationPolicy con métodos create, update, delete.

POST /v2/statements/receipt-upload-url

No se identificó un check explícito de permiso en el controller de upload de recibos.

Riesgo
Cualquier usuario autenticado con acceso al statement puede subir recibos.
Fix
Verificar can:update,statement o equivalente con EditModeAccess.
🟡 Media Severidad

POST /{statement}/sync — Sync individual

StatementPolicy::sync() verifica canBeSynced() && (isAdmin || !isPublished) pero no verifica el permiso statements.sync. Cualquier usuario autenticado con acceso al statement podría sincronizarlo si no está publicado.

Fix
Agregar $user->hasPermissionTo(StatementsSync, $statement->operation_id) en StatementPolicy::sync().

GET /v2/statements/view/yearly — Vista anual

Usa ValidOperation rule que chequea $user->can('view', $operation) ✅ pero no verifica si el user tiene statements.view.approved / unapproved / published.

Fix
Validar el permiso de vista según el estado de los statements en la operación.

GET /v2/statements/all — Listar todos

El filtrado se hace implícitamente vía el trait StatementsQuery según el rol del usuario. No hay una policy explícita.

Observación
El filtrado es funcionalmente correcto pero depende de lógica implícita en el trait. Un bug ahí podría exponer statements incorrectos.

🖥️ Frontend Gaps

Features del frontend que se muestran o ejecutan sin verificar el permiso adecuado.

🔴 Sin verificación alguna

Pestaña "Templates" — sidebar

El sidebar chequea hasPermission(TemplatesView) ✅, pero la backend TemplatePolicy::view() retorna true para cualquier usuario autenticado. El permiso templates.view no está asignado a Internal ni External en el seeder, así que el sidebar los bloquea pero el backend los dejaría pasar si acceden directamente.

Archivo
Sidebar.tsx + TemplatePolicy.php
Fix
Hacer que TemplatePolicy::view() verifique el permiso templates.view en vez de devolver true.

Download monthly/yearly view — botones descarga

StatementView.tsx muestra botones de descarga sin chequear el permiso download.statements. El permiso existe y está asignado a roles, pero nunca se usa ni en front ni en back.

Archivo
StatementView.tsx
Fix
Agregar canDownloadStatements = hasPermission(DownloadStatements, selectedOperation.id) para controlar los botones.

Botón "Publish" en StatementActions.tsx

canPublishStatement() chequea solo status === APPROVED && (accountantId == user.id || user.isAdmin). No usa el permiso statements.publish. Un InternalReportUser que sea accountant ve el botón aunque no tenga el permiso.

Archivo
StatementActions.tsx + useHasPermission.tsx
Fix
Agregar check de hasPermission(StatementsPublish, operationId) en canPublishStatement().
🟡 Verificación parcial

canEditStatement() usa permiso incorrecto

Chequea PermissionNamesEnum.OperationsView en vez de edit-mode.access. operations.view es un permiso de lectura — lo tienen tanto Internal como External, lo que podría mostrar opciones de edición a ExternalReportUser.

Archivo
useHasPermission.tsx
Fix
Cambiar el check a hasPermission(EditModeAccess, operationId) para alinear con el backend.

Permisos ausentes del enum frontend

PermissionNamesEnum.ts tiene 27 permisos — le faltan 3 del backend:

  • StatementsViewPublished — no existe en el enum frontend
  • DeclineStatements — no existe, se delega a statement.canBeDeclinedByCurrentUser
  • SubmitStatements — no existe, se delega a statement.canBeSubmittedByCurrentUser
Impacto
Funcionalmente correcto porque el servidor provee los campos canBe*, pero el front pierde visibilidad explícita del permiso.

🔀 Back ↔ Front Inconsistencies

Casos donde el backend y el frontend verifican cosas diferentes para la misma acción.

🔴 Crítico

User Management — backend usa isAdmin, frontend usa permisos individuales

AcciónBackendFrontend
Crear usuarioadmin middlewareUserManagementCreate
Ver usuariosadmin middlewareUserManagementViewUsers
Editar usuarioadmin middlewareUserManagementEditUsers
Eliminar usuarioadmin middlewareUserManagementDeleteUsers
Asignar rolesadmin middlewareUserManagementAssignPermissions

Impacto: El backend siempre rechazará a non-Admins. Si en el futuro se asignan estos permisos a un InternalReportUser, el frontend los mostraría pero el backend los rechazaría.

Operations Create — backend es isAdmin, frontend chequea operations.create

Backend: ruta POST /operations/ usa middleware admin. Frontend: canCreateOperations = hasPermission(OperationsCreate).

Impacto: Un InternalReportUser tiene operations.create en el seeder pero el backend lo rechazaría. Hoy no genera bug visible (Admin tiene ambos), pero el modelo es inconsistente.

Templates Modify — backend es isAdmin, frontend chequea templates.modify

Backend: TemplatePolicy::create() y delete() verifican isAdmin(). Frontend: hasPermission(TemplatesModify).

Impacto: Si templates.modify se asignara a un non-Admin, el front mostraría los botones pero el back los rechazaría.

🟡 Medio

invoices.visibility.toggle — lógica diferente

CapaLógica
Backend(user.id === accountant_id && !isPublished) || isAdmin
FrontendisAdmin || (isAccountant && hasPermission(InvoicesToggleVisibility) && !isPublished)

El backend no verifica el permiso invoices.visibility.toggle. Si el accountant NO tiene el permiso, el frontend lo bloquea pero el backend lo permitiría.

statements.publish — lógica diferente

CapaLógica
Backend(hasPermission(StatementsPublish) || isAccountant) && isApproved
FrontendisApproved && (accountantId == user.id || isAdmin)

El backend también acepta si el user tiene el permiso (independientemente de ser accountant). El frontend solo chequea accountant/Admin. Un InternalReportUser con statements.publish no vería el botón en el front pero el back sí lo aceptaría.

edit-mode.access — canEditStatement() usa permiso diferente

CapaLógica
BackendhasPermission(EditModeAccess, operationId)
FrontendisAccountant || hasPermission(OperationsView, operationId)

El frontend usa operations.view (permiso de lectura) para decidir si puede editar. operations.view lo tienen tanto Internal como External, potencialmente mostrando opciones de edición a ExternalReportUser.
